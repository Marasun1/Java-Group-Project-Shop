# Java-Group-Project-Shop
## Команда 3  
### Тема для дослідження: Обмеження чистого JDBC та перспективи розвитку підходів до роботи з БД у Java. Критичний аналіз складності ручного керування SQL-запитами та ресурсами.  
### Тема проєкту: Система обліку товарів на складі продуктового магазину.  
База даних — Shop (таблиці Product, quantity, price)

**Актуальність теми.** Автоматизація обліку складських запасів є критичною для продуктових магазинів середнього бізнесу, де обсяг номенклатури перевищує 5 000 найменувань із щоденним оборотом сотень операцій приймання, відвантаження та інвентаризації. Існуючі рішення (1С, CloudFY) часто є надмірно складними та дорогими для локальних мереж. Розробка спеціалізованої системи з використанням сучасних підходів об'єктно-орієнтованого програмування (Java + JDBC) дозволяє створити легковісне рішення з точним відповіданням предметній області та низькими вимогами до hardware.  

**Предметна область.** Система призначена для автоматизації обліку товарно-матеріальних цінностей на складі продуктового магазину. Основні бізнес-процеси: реєстрація приходу від постачальників з серійними номерами/терміном придатності, оформлення відвантажень у торговий зал, інвентаризація з розрахунком розбіжностей, формування звітів про залишки та оборотність. Ключова вимога — підтримка методики FIFO (First In, First Out) для списання товарів та контроль критичних залишків (alert при мінімальній кількості).  

**Структура бази даних.** Реляційна модель даних (PostgreSQL) містить нормалізовані сутності: ROLE (id, name, description — ролі ADMIN/MANAGER/CLERK); USER (id, role_id, email, full_name, password_hash, is_active); PRODUCT (id, sku, name, description); PRICE (id, product_id, amount, valid_from, valid_to, retail_price); QUANTITY (id, product_id, location, qty); RECEIPT (id, product_id, user_id, qty_received, received_at, note, wholesale_price).  

**Зв'язки:** ROLE → USER (1:N), PRODUCT → PRICE (1:N), PRODUCT → QUANTITY (1:N), PRODUCT → RECEIPT (N:1), USER → RECEIPT (N:1)  

**Архітектурне рішення.** Застосунок побудовано за патерном MVC (Model-View-Controller) із чітким розділенням відповідальності. Model — POJO-класи (Product, StockMovement) та DAO-шар (Data Access Object) з чистим JDBC для прямого керування SQL-запитами та транзакціями. View — JavaFX-інтерфейси з FXML-розміткою (окремі форми для приходу, відвантаження, інвентаризації). Controller — обробники подій, що делегують бізнес-логіку сервісному шару (Service Layer). Збірка — Maven (pom.xml з залежностями: javafx-controls, jdbc-postgresql, log4j). Реалізовано пул з'єднань HikariCP для оптимізації доступу до БД.  

**Використані технології.** Backend: Java 17 (LTS), JDBC 4.3 для доступу до PostgreSQL 14. Frontend: JavaFX 17 (FXML + CSS для стилізації), архітектура FXML дозволяє розділити UI-дизайн від логіки. База даних: PostgreSQL з використанням транзакцій рівня READ_COMMITTED, індексів B-tree на зовнішніх ключах (product_id, category_id) для прискорення JOIN-операцій. Система контролю версій: Git. Засоби розробки: IntelliJ IDEA, Scene Builder для візуального редагування FXML. Логування: Log4j2 для фіксації операцій користувачів (аудит).  

**Очікуваний функціонал.** CRUD-операції: створення/редагування/видалення товарів та користувачів; реєстрація надходжень товарів (RECEIPT) з автоматичним оновленням залишків у таблиці QUANTITY. Пошук: динамічна фільтрація товарів за назвою та SKU (LIKE з параметризованими запитами для захисту від SQL-ін'єкцій), за локацією, за датою надходження. Звіти: формування відомостей надходжень за період, аналіз залишків у розрізі локацій (SQL GROUP BY), перегляд актуальних цін товарів з урахуванням дат дії (valid_from / valid_to). Сервісні функції: експорт даних у CSV, попередження про товари з низьким залишком, розрахунок середньої закупівельної ціни через SQL-запити з віконними функціями (ROW_NUMBER() OVER), розмежування доступу відповідно до ролі користувача (ADMIN / MANAGER / CLERK).  

**Висновки.** Розроблено архітектуру автоматизованої системи обліку товарів продуктового складу на базі JavaFX та JDBC. Реалізація патерну MVC забезпечує модульність та тестованість коду. Використання чистого JDBC замість ORM дозволяє оптимізувати продуктивність критичних операцій (пакетна вставка приходів, складні звітні SQL-запити) та мати повний контроль над транзакціями. ER-модель забезпечує цілісність даних та підтримку бізнес-правил FIFO. Очікуваний результат — зниження часу обробки складських операцій на 40% порівняно з ручним обліком у Excel та виключення помилок при розрахунку залишків.
